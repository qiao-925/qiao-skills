# 来源质量控制

> 约束 Agent 在依赖外部知识时优先使用高可信、可溯源、相对权威的一手或近一手来源，并按来源质量校准结论力度。

## 核心原则

### 1. 来源分级 —— P0 / P1 / P2

- **P0（一手核心来源）**：官方文档、API reference、release notes、changelog、官方博客/工程博客、官方 GitHub 仓库/README/设计说明、核心维护者原始说明、标准文本/RFC/原论文/原始数据。
- **P1（高可信补充来源）**：大厂技术架构文章、顶级会议/权威研究论文、核心维护者访谈、高可信机构报告、高质量综述/技术白皮书。
- **P2（二级参考来源）**：高质量技术分析文章、社区总结、第三方测评、非原始的经验贴/教程/博客二次解读。

### 2. 结论力度校准 —— 来源越弱，结论越保守

| 来源情况 | 建议结论力度 |
|----------|-------------|
| 多个 P0 一致 | 可给较强结论，但仍应说明边界 |
| 单个 P0 或 P0 + P1 | 可给倾向性结论，并说明假设 |
| 只有 P1 | 给保守建议，尽量补充交叉验证 |
| 只有 P2 | 只能给弱结论，并明确不确定性 |

### 3. 事实、推断、建议分开 —— 不混写

- 事实：必须可追溯到具体来源。
- 推断：必须显式写明"这是基于资料的推断"。
- 建议：必须说明为什么当前建议优于替代路径。

### 4. 高时效内容重新验证 —— 不靠缓存

- 若结论依赖版本号、API 行为、维护状态等时效性信息，优先重新核对。
- 不要用训练数据中的记忆替代实时查证。

## AI Agent 行为要求

### 默认适用场景

| 场景 | 最低要求 | 不该做什么 |
|------|----------|------------|
| 资料收集 | 标注每条关键结论的来源等级 | 把二手总结当一手来源 |
| 技术调研 | 关键结论用 P0 支撑 | 只看社区博客就给结论 |
| 方案比较 | 不同维度标注来源等级 | 用第三方横评代替原始能力说明 |
| API / SDK 使用 | 先查官方文档，再补 issue | 靠模型记忆猜参数名 |

### 默认执行方式

1. 先明确当前到底在查什么。
2. 按来源等级优先查 P0，不够再补 P1/P2。
3. 关键结论绑定来源等级和版本条件。
4. 区分事实、推断、建议，不混写。
5. 来源不足时主动降低结论力度。

## 判断标准

- 当前结论的主支撑来源属于哪一级。
- 是否存在更高等级来源尚未核对。
- 是否因为来源偏弱而需要收缩语气。
- 是否已保留出处、时间点和上下文。

## 反模式

- 靠模型记忆猜 API 参数名或调用方式。
- 只看 README，不看 release notes 和 migration guide。
- maintainer 已说明弃用，但仍沿用旧示例。
- 用 star 数或热度替代维护状态判断。
- 把来源越多等同于质量越高。
- 用二手总结主导整篇调研结论。

## 工程场景来源验证

> 用于编码、API、SDK、版本迁移、开源库接入与维护状态判断。

### 何时读取

- 新接第三方库、SDK、框架或服务
- 参数名、方法名、返回值、配置项不确定
- 版本升级、迁移、弃用、兼容性判断
- 报错明显指向 API 使用方式不对
- 评估开源库是否值得接入、是否还在维护

### 工程来源优先级

**API / SDK / 框架行为：**

1. 官方文档 / API reference
2. 官方 release notes / migration guide / changelog
3. 官方示例 / 仓库 README / 设计说明
4. maintainer comment / 高信号 issue / discussion
5. 社区教程或第三方分析

**开源库接入与维护状态：**

1. 官方仓库 README / docs
2. release notes / changelog / roadmap
3. maintainer comment、design note、high-signal issue
4. 社区总结与第三方文章

检查重点：最近 release/commit/issue 响应是否正常，是否有明确迁移说明，README 与当前版本是否一致，是否存在长期未处理的关键缺陷。

### 最低执行流程

1. **收敛未知点**：先明确在查 API 用法、版本差异还是维护状态。
2. **补全最小上下文**：库名、版本号、报错信息、当前使用方式。
3. **先读 P0**：官方文档、API reference、release notes。若已足够则不扩大搜索。
4. **结论绑定版本与来源**：写清依据来源、适用版本、未验证假设。
5. **实现后验证真实行为**：最小复现、单元测试或运行日志检查。

### 关键判断门槛

- **只看文档就够**：标准 API 用法、文档和示例一致、无异常报告。
- **必须看 release notes**：版本升级、弃用提示、示例代码冲突。
- **必须看 maintainer / issue**：文档明显过期、真实行为未被解释、判断长期接入价值。

## 研究与综合场景来源整合

> 用于资料收集、技术调研、产品与方案比较、研究总结、长文档写作，以及多 agent 并行证据整合。

### 研究来源优先级

**资料收集 / 技术调研：**

1. 官方材料、原始数据、原论文、标准文本
2. 权威机构报告、顶级研究论文、核心维护者说明
3. 高质量综述和二级分析

**产品 / 工具 / 方案比较：**

1. 官方文档、产品说明、限制页、价格页、更新日志
2. 官方博客、发布会、工程文章
3. 高可信第三方评测或横评

### 文档合成流程

1. **先定主线**：明确核心问题、输出类型、读者最需要带走的判断。
2. **按来源等级收集证据**：做证据表，标注来源等级和是否属于推断。
3. **先组织判断，再组织章节**：围绕判断而非来源顺序。
4. **区分事实、推断与建议**：推断必须显式标注。
5. **统一终稿**：多 agent 并行时，主控统一去重、校准术语、合并冲突、调整结论力度。

### 文档输出契约

正文最低要求：逻辑连贯、有明确主线、关键结论能回指来源、不把材料卡片直接堆成章节。

比较类输出：比较维度明确、不同前提写成条件化结论、第三方横评指出局限。

### 并行执行

- 主控先定义边界、目录、来源等级和证据标准
- worker 并行收集不同主题的材料
- 统一回收证据包，不直接拼接正文
- 主控统一编排、去重、补过渡、校准结论力度

### 可重复执行最小清单

每次执行至少保留：任务目标、检索范围、主要来源等级、关键来源链接、纳入/剔除理由、执行时间点、事实与推断的边界。
