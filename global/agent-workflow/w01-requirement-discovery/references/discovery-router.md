# 问题分类与视角选择清单

## 1. 目的

本清单用于在 `w01` 触发后，帮助 Agent 先判断当前问题属于哪一类，再选择最合适的分析视角。

默认规则：
- 先分类，再选模型。
- 默认只启用 `1 个主视角 + 1 个补充视角`。
- 所有视角最终都要回到 `ROI + 价值密度` 做统一收敛。

## 2. 问题分类

| 当前问题 | 常见信号 | 主视角 | 补充视角 |
|---------|----------|--------|----------|
| 用户任务不清 | "用户到底要什么？"、"这个产品该解决什么问题？" | `JTBD` | `User Journey` 或 `角色辅助` |
| 流程卡点明显 | "哪一步最卡？"、"为什么转化差？" | `User Journey` | `JTBD` |
| 候选很多难排序 | "哪些必须先做？"、"哪些只是加分项？" | `Kano` | `Strategic Fit` |
| 新方向不确定 | "值不值得投？"、"要不要先验证？" | `Assumption Mapping` | `Strategic Fit` 或 `JTBD` |
| 担心偏离主线 | "会不会越做越散？"、"符不符合方向？" | `Strategic Fit` | `Kano` 或 `角色辅助` |

如果无法明确归类，先问自己：
- 当前最大的问题是"不知道解决什么"还是"不知道先做什么"？
- 当前最大的风险是"选错问题"还是"投入过大但验证太慢"？

## 3. 路由 checklist

1. 当前最主要的困惑是什么？
2. 当前证据主要来自哪里？
3. 哪个视角最能解释这个困惑？
4. 是否需要第二个视角来防止偏差？
5. 候选方向形成后，是否已回到 `ROI + 价值密度` 做统一收敛？

## 4. 证据类型提示

| 证据类型 | 更适合的视角 |
|---------|--------------|
| 用户目标模糊、访谈片段、需求描述零散 | `JTBD` |
| 流程步骤、转化漏斗、使用过程抱怨 | `User Journey` |
| 候选需求列表、功能讨论、优先级争议 | `Kano` |
| 新方向、实验想法、关键假设未确认 | `Assumption Mapping` |
| 产品目标、核心用户、路线图边界 | `Strategic Fit` |

## 5. 补充视角使用规则

- 只有在主视角无法充分解释问题时，才启用补充视角。
- 补充视角的作用是修正偏差，不是重新开一套独立分析。
- 若补充视角没有提供新增信息，应立即收敛。

## 6. 角色辅助的定位

`角色辅助` 不是主视角，只在以下情况作为补充技术使用：
- 需要判断某个方向对谁更重要。
- 需要判断某个方向是否把成本转移给其他角色。
- 需要补充用户、开发者、维护者三个立场之一的盲区。

详见：`references/role-play-method.md`

## 7. 不适合继续 discovery 的情况

- 用户已经给出明确、可执行的功能需求。
- 用户要的是技术实现、代码修复或方案落地。
- 当前争议已经不是"该做什么"，而是"如何设计实现"。

此时应转入 `w02-task-planning` 或对应执行流程。

## 8. 最小输出要求

每次使用路由时，输出至少说明：
- 当前问题类型
- 主视角
- 补充视角
- 选择理由
- 最终如何收敛到 `ROI + 价值密度`

## 9. 最小使用示例

```text
当前问题类型：候选很多难排序
主视角：Kano
补充视角：Strategic Fit
选择理由：当前已有候选列表，核心问题是区分基础项与加分项，同时避免偏离主线。

候选方向 A：先补基础体验缺口
- Kano：属于基础项
- Strategic Fit：直接服务当前核心用户
- ROI / 价值密度：高
```

## 10. 反模式

- 一次性并行启用多个视角，导致输出发散。
- 没有问题分类就直接套模型。
- 把 `ROI` 当成唯一视角，从而跳过问题发现。
- 把角色辅助当成主模型。
- 为了显得专业而强行跑完整 checklist。
