# 触发信号详细识别规则

## 1. 触发条件识别

### 明确询问类

| 关键词 | 示例 |
|--------|------|
| "有哪些功能" | "这个项目有哪些功能可以开发？" |
| "应该做什么" | "我应该做什么？"、"接下来应该做什么？" |
| "优先级" | "优先级是什么？"、"哪些功能优先级高？" |
| "下一步" | "下一步做什么？"、"接下来呢？" |
| "梳理" | "帮我梳理一下任务"、"梳理一下功能" |
| "分析需求" | "分析一下需求"、"看看有什么需求" |
| "探索" | "探索一下方向"、"看看有什么可以做的" |

### 迷茫表达类

| 语气特征 | 示例 |
|---------|------|
| 不确定 | "不知道"、"不太清楚"、"有点迷茫" |
| 询问式 | "？"、"呢？"、"吗？" |
| 求助式 | "帮我看看"、"给点建议"、"有什么想法" |

## 2. 触发后的问题分类线索

触发 `w01` 后，不要直接套固定模板，应继续判断当前问题更像哪一类：

| 线索 | 常见表达 | 优先路由 |
|------|----------|----------|
| 用户任务不清 | "用户到底要什么？"、"这个产品该解决什么问题？" | `JTBD` |
| 流程卡点明显 | "转化低"、"体验哪里别扭"、"流程哪一步最卡" | `User Journey` |
| 候选方向太多 | "哪些先做？"、"哪些是必须的？" | `Kano` |
| 不确定性高 | "值不值得做？"、"是不是该先验证？" | `Assumption Mapping` |
| 担心偏离主线 | "符不符合方向？"、"会不会越做越散？" | `Strategic Fit` |

若仍不确定，先进入 `references/discovery-router.md` 按 checklist 选择视角。

## 3. 上下文判断

### 触发场景

- 项目启动初期，用户不确定方向
- 完成一个功能后，用户询问下一步
- 用户表达对项目方向的困惑
- 用户要求"全面分析"、"整体规划"

### 不触发场景

- 用户已有明确功能需求（应进入 `w02-task-planning`）
- 用户询问技术实现细节（应进入对应技术分析或实现流程）
- 用户询问代码问题（应直接回答）

## 4. 判断流程

```
用户输入
  ↓
检测关键词/语气
  ↓
判断是否表达迷茫/询问方向
  ↓
是 → 触发 w01-requirement-discovery
  ↓
按问题线索进入 discovery-router
否 → 继续其他流程
```

## 5. 示例

### ✅ 应该触发

- "这个项目应该做什么功能？"
- "帮我梳理一下有哪些可以做的"
- "有点迷茫，不知道下一步该做什么"
- "当前最应该优先实现什么？"
- "这个方向值不值得做，还是该先验证？"
- "这些需求里哪些是基础项，哪些只是加分项？"

### ❌ 不应该触发

- "实现一个用户登录功能"（明确需求，应进入 w02-task-planning）
- "这个函数怎么优化？"（技术问题，直接回答）
- "帮我修复这个bug"（明确任务，直接执行）
