# 角色扮演方法与实践

## 1. 核心原理

**让AI假设自己是目标用户群体，从该视角列出"必须实现的高ROI功能"**

这是发现需求盲区的有效方法，而非完全依赖人工需求分析。

## 2. 角色类型

### 2.1 目标用户角色

**适用场景**：面向最终用户的产品

**角色特征**：
- 关注使用体验、功能完整性
- 从"我需要什么"的角度思考
- 重视易用性、性能、稳定性

**分析维度**：
- 核心使用场景
- 痛点与期望
- 功能完整性
- 用户体验

**示例**：
- RAG系统 → "作为知识工作者，我需要快速检索、准确答案、多格式支持"
- 开发工具 → "作为开发者，我需要快速集成、清晰文档、稳定API"

### 2.2 开发者角色

**适用场景**：技术产品、开发工具、基础设施

**角色特征**：
- 关注技术架构、可维护性、扩展性
- 从"如何实现"、"如何维护"的角度思考
- 重视代码质量、测试覆盖、文档完整性

**分析维度**：
- 架构合理性
- 代码可维护性
- 测试与文档
- 性能与扩展性

**示例**：
- 后端系统 → "作为开发者，我需要清晰的模块划分、完善的测试、良好的文档"
- 框架工具 → "作为使用者，我需要简单API、丰富示例、详细文档"

### 2.3 维护者角色

**适用场景**：长期维护的项目

**角色特征**：
- 关注长期可维护性、技术债务
- 从"如何持续改进"的角度思考
- 重视代码质量、架构清晰度、重构空间

**分析维度**：
- 技术债务
- 重构机会
- 性能瓶颈
- 可观测性

**示例**：
- 成熟项目 → "作为维护者，我需要清理技术债务、优化性能、提升可观测性"

## 3. 角色选择策略

### 3.1 根据项目类型推荐

| 项目类型 | 推荐角色 | 理由 |
|---------|---------|------|
| 面向用户的产品 | 目标用户 | 关注用户价值 |
| 开发工具/框架 | 开发者 | 关注开发者体验 |
| 基础设施/库 | 开发者 + 维护者 | 关注技术质量 |
| 长期维护项目 | 维护者 | 关注可持续性 |

### 3.2 多角色并行

对于复杂项目，可以**并行使用多个角色**，发现不同维度的需求：

```
角色1（用户）→ 功能需求列表
角色2（开发者）→ 技术需求列表
角色3（维护者）→ 改进需求列表
  ↓
合并去重，按ROI排序
```

## 4. 执行方法

### 4.1 角色扮演指令

```
假设你是 [角色描述]，正在使用/开发/维护这个项目。

从你的视角，列出：
1. 当前最缺失的、必须实现的高ROI功能
2. 每个功能的影响范围和实现成本
3. 优先级排序

要求：
- 聚焦高ROI（影响大、成本低）
- 从实际使用/开发/维护场景出发
- 避免理想化，考虑现实约束
```

### 4.2 输出要求

每个功能应包含：
- **功能描述**：清晰说明是什么
- **为什么需要**：从角色视角说明必要性
- **影响范围**：影响哪些用户/场景
- **实现成本**：开发时间、复杂度评估
- **优先级**：P0/P1/P2

## 5. 实践示例

### 示例：RAG系统

**角色**：知识工作者（目标用户）

**角色扮演**：
> "作为知识工作者，我每天需要从大量文档中快速找到答案。当前系统最缺失的是：
> 
> 1. **多轮对话上下文** - ROI: ⭐⭐⭐⭐⭐
>    - 影响范围：所有需要深入讨论的场景
>    - 实现成本：中（需要会话管理）
>    - 优先级：P0
> 
> 2. **文档预览与高亮** - ROI: ⭐⭐⭐⭐
>    - 影响范围：需要查看原文的场景
>    - 实现成本：低（前端展示）
>    - 优先级：P1"

## 6. 注意事项

### 6.1 避免理想化

- 不要列出"完美但不可实现"的功能
- 考虑项目当前阶段、资源约束
- 聚焦"最小可行"的高ROI功能

### 6.2 聚焦高ROI

- 优先发现"影响大、成本低"的功能
- 避免"影响小、成本高"的功能
- 考虑实现可行性和时间成本

### 6.3 结合项目现状

- 分析项目当前状态
- 识别最紧迫的缺失
- 考虑技术栈和架构约束
