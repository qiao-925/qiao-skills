# User Journey 视角

## 1. 目的

`User Journey` 用来回答：用户在哪一步最卡，哪个流程断点最值得优先修复。

它适合发现转化、体验、流程协作中的真实卡点。

## 2. 什么时候用

- 用户已经能描述使用流程，但不知道哪一步最影响结果。
- 团队怀疑有转化、留存、体验断点。
- 需要从流程角度判断"先修哪里更有杠杆"。

## 3. 不适合什么时候用

- 还不知道用户到底要完成什么任务。
- 当前主要问题是战略主线或高不确定性方向验证。

## 4. 核心问题

1. 当前要分析的是哪一段旅程？
2. 旅程可分成哪几个关键阶段？
3. 每个阶段的主要摩擦点是什么？
4. 哪个摩擦点最影响成功率、体验或推进效率？
5. 修复哪个环节最可能带来最大改善？

## 5. 最小判断步骤

1. 明确旅程起点和终点。
2. 把过程拆成 4-6 个关键阶段。
3. 标出最明显的摩擦、等待、流失或认知负担。
4. 选择最值得优先修复的 1-2 个节点。
5. 回到 `ROI + 价值密度` 判断先修哪一个。

## 6. 推荐输出

```text
主视角：User Journey
- 当前旅程：从……到……
- 关键卡点：……
- 候选方向：先修……，因为……
```

## 7. 反模式

- 一上来就画完整旅程，忽略当前真正要解决的范围。
- 把每个环节都当成同等优先级。
- 只描述页面或功能，不描述用户动作与阻塞。
- 没有回到统一评估层就直接下结论。
