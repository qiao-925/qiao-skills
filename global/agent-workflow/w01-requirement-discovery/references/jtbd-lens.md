# JTBD 视角

## 1. 目的

`JTBD` 用来回答：用户真正想完成什么任务，当前为什么还没顺利完成。

它适合把注意力从"要不要做某个功能"拉回到"用户到底想取得什么进展"。

## 2. 什么时候用

- 用户需求表述停留在功能层。
- 团队列了很多点子，但不知道核心问题是什么。
- 需要区分"表面诉求"和"真实任务"。

## 3. 不适合什么时候用

- 用户已经给出明确、可执行的落地需求。
- 当前主要问题不是任务不清，而是流程卡点或战略取舍。

## 4. 核心问题

1. 用户在什么情境下需要这个产品或能力？
2. 用户真正想完成的任务是什么？
3. 用户现在用什么替代方案在完成这个任务？
4. 当前最大的摩擦点是什么？
5. 什么结果会让用户觉得这件事"被完成了"？

## 5. 最小判断步骤

1. 识别目标用户与使用情境。
2. 用一句话描述 job：`当……时，我想……，以便……`。
3. 找出阻碍这个 job 的主要摩擦点。
4. 围绕 job 和摩擦点提出 2-3 个候选方向。
5. 回到 `ROI + 价值密度`，判断当前先做哪个更划算。

## 6. 推荐输出

```text
主视角：JTBD
- 用户任务：当……时，我想……，以便……
- 当前摩擦：……
- 候选方向：……
```

## 7. 反模式

- 把功能名直接当成 job。
- 只写用户画像，不写用户要完成的任务。
- 忽略使用情境，只讨论抽象愿望。
- 找到 job 后立刻跳进大而全方案。
