# Kano 视角

## 1. 目的

`Kano` 用来回答：候选需求里哪些是基础项，哪些是线性增益项，哪些只是加分项。

它适合在候选方向较多时，避免把"锦上添花"误判成"当前必须先做"。

## 2. 什么时候用

- 已经有一批候选功能或方向，但优先级争议较大。
- 需要区分"不做会出问题"和"做了更惊喜"。
- 需要防止加分项盖过基础项。

## 3. 不适合什么时候用

- 当前还没搞清楚用户任务是什么。
- 候选方向很少，且问题已经明显是战略或风险验证类。

## 4. 核心问题

1. 哪些能力缺失会直接导致不满或不可用？
2. 哪些能力会随质量提升带来稳定收益？
3. 哪些能力属于惊喜项，但不是基础必需？
4. 当前是否存在基础项没补齐，却想先做加分项？

## 5. 最小判断步骤

1. 列出当前候选需求。
2. 粗分为：基础项、线性增益项、惊喜项、低相关项。
3. 优先检查基础项是否存在明显缺口。
4. 若多个候选都成立，再回到 `ROI + 价值密度` 做轻量排序。

## 6. 推荐输出

```text
主视角：Kano
- 基础项：……
- 线性增益项：……
- 惊喜项：……
- 当前建议：先补基础项 / 先做线性收益最高项
```

## 7. 反模式

- 没有候选列表就强行做 Kano 分类。
- 把所有功能都归为基础项。
- 用 Kano 直接替代 ROI 判断。
- 没有用户证据时，把分类说成确定事实。
