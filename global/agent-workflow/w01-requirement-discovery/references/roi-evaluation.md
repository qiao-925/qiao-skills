# ROI 与价值密度的统一评估层

## 1. 目的

不管主视角是 `JTBD`、`User Journey`、`Kano`、`Assumption Mapping` 还是 `Strategic Fit`，最终都应回到同一层比较：
- 这个方向整体上值不值得做
- 这个方向当前阶段先做是否划算
- 这个方向为什么现在做，为什么暂不做

## 2. 关系定义

- **ROI**：回答"整体上值不值得做"，关注投入与回报是否匹配。
- **价值密度**：回答"当前阶段先做哪个更划算"，关注单位时间、单位复杂度、单位认知负担下的推进价值。

二者不是替代关系，而是同一判断原则在不同抽象层上的表达：
- ROI 是总纲。
- 价值密度是需求发现阶段的补充视角。

## 3. 在 w01 中的位置

- 各个视角负责发现问题、解释问题或区分问题。
- `ROI + 价值密度` 负责统一收敛与轻量排序。
- 它不是单独替代其他视角的主入口，而是跨视角的公共比较层。

## 4. 在 w01 中如何使用

默认按以下顺序判断：

1. **先看主视角结论**：当前真正的问题是什么，候选方向为什么成立。
2. **再看 ROI**：这个方向整体上是否值得投入。
3. **再看价值密度**：如果有多个都值得做，当前先做哪个更容易以较小投入带来较大推进。
4. **最后做轻量排序**：只给出足够帮助用户收敛的比较，不追求伪精确。

## 5. 核心概念

可以把 ROI 和价值密度简单理解为：

```text
ROI ≈ 有效回报 / 总投入
价值密度 ≈ 当前阶段有效价值 / 新增复杂度
```

这里的表达是启发式的，不是要求在 discovery 阶段做严格量化计算。

一个方向的价值密度通常更高，当它同时满足：
- 能直接解决当前阶段的真实问题。
- 能在较短周期内被验证。
- 不需要引入明显超前设计或全局性复杂度。
- 能带动后续决策、实现或协作继续推进。

## 6. 轻量判断维度

### 6.1 预期价值

重点看三个问题：
- 这个方向是否明显改善用户结果、开发效率或维护质量？
- 它是否命中当前阶段的核心问题，而不是边缘优化？
- 它是否能减少后续反复讨论、返工或停滞？

### 6.2 投入与新增复杂度

重点看三个问题：
- 实现是否需要引入新的抽象层、配置层或系统耦合？
- 影响范围是局部可控，还是会扩散到多个模块和流程？
- 需要的前置条件是否过多？

### 6.3 验证速度

重点看两个问题：
- 做完后能否在短周期内知道值不值得？
- 能否在不承诺大规模投入的前提下先验证方向？

### 6.4 不确定性

重点看两个问题：
- 关键假设是否清楚？
- 是否依赖尚未确认的用户、数据、接口或组织条件？

## 7. 推荐判断方式

默认使用定性判断即可：

| 维度 | 建议标记 |
|------|----------|
| 预期价值 | 高 / 中 / 低 |
| 新增复杂度 | 低 / 中 / 高 |
| 验证速度 | 快 / 中 / 慢 |
| 不确定性 | 低 / 中 / 高 |

一般可以这样收敛：
- **优先推进**：价值高，复杂度低到中，验证快，不确定性可控。
- **值得保留但延后**：价值高，但复杂度或不确定性偏高。
- **暂不建议**：价值不够集中，且复杂度明显偏高。

如果需要一句话概括：
- ROI 决定"值不值得做"。
- 价值密度帮助判断"现在先做哪个更划算"。

## 8. 什么时候可以用更明确的优先级

如果用户明确要求排序，可以使用 `P0 / P1 / P2` 或 `现在做 / 稍后做 / 暂不做`。

但应避免：
- 复杂权重公式
- 过多小数分数
- 为了排序而硬凑评分项

## 9. 例外情况

即使价值密度表面上不高，以下情况也可以上调优先级：
- 阻塞当前主线推进的问题
- 会持续制造返工的结构性缺口
- 用户已明确承诺要做、且决策成本高于实现成本
- 能显著提升后续判断清晰度的基础工作

## 10. 推荐输出

```text
[方向名]
- 主视角结论：它解决了什么关键问题
- ROI：高
- 价值密度：高
- 为什么现在做：整体值得做，且能直接缓解当前最主要的卡点
- 成本/复杂度：低到中
- 主要风险：需要确认一个关键假设
```

如果一句话就能说明白，不必强行套格式。

## 11. 反模式

- 用 ROI 单独取代问题发现视角。
- 没有证据却给出精确分数和复杂权重。
- 只说"值得做"，却不解释它解决的核心问题是什么。
- 把 `ROI + 价值密度` 和 `Kano`、`JTBD` 等模型混写成同一种分析。
