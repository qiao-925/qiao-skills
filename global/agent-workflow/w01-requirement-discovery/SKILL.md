# 需求发现规范

> 用问题分类 + 多视角策略路由，帮助用户找到当前最值得推进的下一步。

---

## ⚠️ 核心强制要求

### 自动触发条件

检测到以下信号时**必须触发**：
- 用户询问："有哪些功能？"、"应该做什么？"、"优先级？"、"下一步？"
- 用户表达迷茫、不确定的语气
- 用户要求"梳理任务"、"分析需求"、"探索方向"

### 核心价值观

- **ROI 优先**：优先考虑整体收益高、投入合理、值得推进的方向。
- **价值密度补充**：在 ROI 判断基础上，优先识别单位时间、单位复杂度、单位认知负担下最值得先做的方向。
- **先分类再选模型**：先判断当前问题属于哪一类，再选择最合适的分析视角，而不是把所有方法一起跑一遍。
- **1 个主视角 + 1 个补充视角**：默认只启用一个最有解释力的主视角，必要时再补一个辅助视角。
- **少而准优于多而全**：默认给出少量高质量候选，不追求一次列全所有可能功能。
- **先澄清核心目标，再扩张功能面**：先找到当前阶段最关键的用户价值或项目推进阻塞点。
- **发现先于规划**：本阶段负责启发和收敛方向，不提前展开详细方案设计、任务拆解或实现细节。

### 分层抽象

- **主流程层**：`SKILL.md` 只定义触发条件、总流程、路由原则、输出契约。
- **路由层**：`references/discovery-router.md` 负责问题分类、策略选择和启用边界。
- **视角层**：各个 `*-lens.md` 负责单一分析视角的适用场景、核心问题和反模式。
- **统一评估层**：`references/roi-evaluation.md` 负责将不同视角的输出统一收敛到 `ROI + 价值密度`。

### 最小发现流程

1. **项目分析**：快速理解项目定位、当前状态、核心价值和现实约束。
2. **问题分类**：判断当前更像是问题发现、流程卡点、候选排序、假设验证，还是战略校准。
3. **策略路由**：选择 1 个主视角，必要时再选 1 个补充视角。
4. **方向生成**：提出 2-3 个候选方向，聚焦"为什么现在做"。
5. **统一收敛**：用 `ROI + 价值密度` 做轻量比较，并标出主要风险和待验证点。
6. **启发输出**：以对话方式帮助用户确认一个方向，再衔接后续规划。

## AI Agent 行为要求

### 触发时执行

1. **先判断当前主要问题**：用户任务不清、流程卡点、优先级混乱、不确定性高，还是主线漂移。
2. **按路由清单选视角**：根据 `references/discovery-router.md` 选择主视角和补充视角。
3. **使用视角生成候选**：通过选中的视角提出 2-3 个候选方向，而不是直接列功能清单。
4. **统一回到 ROI + 价值密度**：对候选方向做轻量比较，明确为什么现在做、为什么暂不做。
5. **区分证据与推断**：明确哪些判断来自用户输入或项目现状，哪些属于 Agent 推断。
6. **帮助收敛**：引导用户从候选中确认一个方向，而不是继续无限发散。

### 输出原则

- 默认使用对话式启发输出，不生成长文档。
- 默认给出 2-3 个候选方向，除非用户明确要求更完整清单。
- 必须说明本次使用的主视角、补充视角和选择理由。
- 星级、分数、表格、P0/P1/P2 仅在能提升清晰度时使用，不作为固定动作。
- 角色切换是辅助技术，只在能强化当前视角判断时使用，不单独取代主视角。
- 若证据不足，必须显式标记为推断，不得伪装成数据结论。
- 如果用户已经明显偏向某个方向，应优先做确认和补充判断，而不是重新发散全部候选。
- 若发现用户其实已有明确需求，应停止 discovery，转入 `w02-task-planning` 或直接执行相关流程。

### 推荐呈现方式

```
💡 需求发现

- 当前问题类型：[问题发现 / 流程卡点 / 候选排序 / 假设验证 / 战略校准]
- 主视角：[视角名]
- 补充视角：[视角名 / 无]
- 选择理由：[为什么这两个视角最适合当前问题]

1. [方向名]
   - 解决的问题：[当前缺口或阻塞点]
   - 判断依据：[来自当前视角的关键观察]
   - ROI / 价值密度：[高 / 中 / 低，或一句话说明]
   - 风险 / 待验证：[一句话说明]

2. [方向名]
   - 解决的问题：...
   - 判断依据：...
   - ROI / 价值密度：...
   - 风险 / 待验证：...

建议：确定方向后，再进入 /w02-task-planning 做详细规划
```

### 衔接 w02-task-planning

用户确认方向后，提示：
```
已确认方向：[方向名]，是否执行 /w02-task-planning 进行详细规划？
```

### 与 W00 协同（自动 + 手动）

- 方向确认且预计跨会话执行时，由主流程阶段完成 issue 初始化/绑定，再调用 `w00-workflow-checkpoint` 写首个 checkpoint。
- 用户手动调用 `/w00-workflow-checkpoint` 时，聚焦存档/读档，不负责初始化。
