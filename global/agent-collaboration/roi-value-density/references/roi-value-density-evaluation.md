# ROI 与价值密度统一判断框架

## 1. 目的

本框架用于帮助 Agent 和用户在当前上下文里回答三个问题：

- 这个动作整体上值不值得做
- 这个动作现在先做是否划算
- 如果不现在做，为什么应该延后

## 2. 关系定义

- **ROI**：回答“整体值不值得做”，关注总回报与总投入是否匹配。
- **价值密度**：回答“当前先做哪个更划算”，关注单位时间、单位复杂度、单位认知负担下的有效推进价值。

二者不是替代关系，而是同一决策原则在不同抽象层上的表达：

- ROI 决定是否值得进入候选池。
- 价值密度决定候选池里谁更适合现在推进。

## 3. 默认使用顺序

1. 先定义当前目标、阶段和阻塞点。
2. 再判断每个候选的 ROI。
3. 只对 ROI 成立的候选继续比较价值密度。
4. 给出轻量收敛结论，而不是追求伪精确排序。

## 4. 启发式表达

可以把两者简单理解为：

```text
ROI ≈ 有效回报 / 总投入
价值密度 ≈ 当前阶段有效价值 / 新增复杂度
```

这里只是启发式表达，不要求严格量化。

## 5. 核心比较维度

### 5.1 预期价值

- 是否命中当前真正的问题，而不是边缘优化
- 是否改善用户结果、交付效率或维护质量
- 是否减少返工、卡点或持续讨论成本

### 5.2 总投入与新增复杂度

- 是否需要新增抽象层、配置层或系统耦合
- 影响范围是局部可控还是会扩散
- 是否依赖过多前置条件

### 5.3 验证速度

- 多快能知道它是否有效
- 能否先做小步验证，而不是一上来大投入

### 5.4 不确定性

- 关键假设是否清楚
- 是否依赖未确认的用户、数据、接口或组织条件

## 6. 高价值密度的典型信号

一个候选通常更适合“现在做”，当它同时满足以下几项：

- 能直接缓解当前最主要的阻塞
- 验证周期短
- 新增复杂度低到中
- 做完后能显著提升下一步判断或执行效率

## 7. 可上调优先级的特殊情况

以下情况即使表面上不够“短平快”，也可以上调优先级：

- 当前主线被它阻塞
- 它持续制造返工
- 它能显著提升后续判断清晰度
- 决策拖延成本高于实现成本

## 8. 推荐输出

```text
[候选名]
- 当前问题：它解决了什么关键问题
- ROI：高 / 中 / 低
- 价值密度：高 / 中 / 低
- 为什么现在做：一句话解释当前推进价值
- 为什么不先做别的：一句话解释比较结果
- 主要风险：一句话说明待验证点
```

如果一句话就能说清，不必强行套格式。

## 9. 反模式

- 只说“值得做”，却不解释解决了什么问题
- 只看实现成本，不看回报质量
- 只看短期速度，不看主线价值
- 没有证据却给精确分数和复杂权重
