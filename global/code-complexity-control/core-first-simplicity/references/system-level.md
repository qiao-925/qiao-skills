# 系统级：复杂度预算与架构简化

## 核心原则

系统设计应先保证主链路稳定，再扩展非核心能力。

## 复杂度预算

只有在以下场景可增加系统复杂度：

- 性能瓶颈有明确监控证据。
- 可靠性目标无法通过简单方案达成。
- 外部约束（合规/安全）必须引入额外机制。

## 架构简化策略

- 优先单一主路径，避免多套并行机制。
- 优先可替换边界，避免全局可配置泛化。
- 优先收敛依赖数量，减少外部耦合点。

## 反模式

- 为未来规模预设过多层次与扩展点。
- 未验证需求就引入插件化/微服务化。
- 把“可选能力”升级为“全局必经路径”。

## 检查清单

- [ ] 主链路是否清晰且可单独解释。
- [ ] 新增系统组件是否有明确必要性证据。
- [ ] 是否存在可删除而不影响核心目标的层。
