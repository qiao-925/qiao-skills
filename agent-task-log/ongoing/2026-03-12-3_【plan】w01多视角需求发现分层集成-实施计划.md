## 背景与目标

当前 `w01-requirement-discovery` 已具备基础 discovery 流程，以及 `ROI + 价值密度` 的判断框架，但视角仍然偏单一。目标不是把所有产品方法论堆进一个 skill，而是在保持可执行性的前提下，扩展为“路由式多视角需求发现框架”：

1. 保留 `ROI + 价值密度` 作为统一评估层。
2. 新增多种主流发现视角，提升问题识别、优先级判断、风险验证与战略对齐能力。
3. 让 Agent 先做问题分类，再自主选择最合适的 `1 个主视角 + 1 个补充视角`。
4. 用统一输出契约保证分析结果可比较、可解释、可复盘。

---

## 设计原则

### 1. 分层抽象

- **L1 主流程层**：`SKILL.md` 只定义触发条件、总流程、路由原则、输出契约。
- **L2 路由层**：用单独 reference 定义问题分类、视角选择 checklist、启用边界。
- **L3 视角层**：每个模型单独一个 reference，按统一模板组织。
- **L4 统一评估层**：所有视角最终回到 `ROI + 价值密度` 做轻量比较与收敛。

### 2. 复杂度控制

- 默认只启用 `1 个主视角 + 1 个补充视角`，禁止一次性并行运行过多模型。
- 第一阶段只接入最小可用视角集，不追求全量收录。
- 数字打分、表格、排序仅在输入信息足够时使用，不做伪精确分析。

### 3. 渐进式披露

- `SKILL.md` 保持短而强，只保留规则与路由。
- 模型细节下沉到 `references/`。
- 同一规则只维护一处，避免主文件和 references 重复。

## 第一阶段集成范围

### 必选统一层

- `ROI + 价值密度`：作为总判断框架与统一收敛层。

### 第一阶段推荐视角集

- `JTBD`：发现用户真正要完成的任务。
- `User Journey`：发现流程卡点、摩擦点、流失点。
- `Kano`：区分基础需求、期望需求、惊喜需求。
- `Assumption Mapping`：识别高不确定性方向，决定先验证什么。
- `Strategic Fit`：判断候选方向是否服务产品主线。

### 暂缓到第二阶段

- `RICE`：与 `ROI` 重叠较高，待第一阶段稳定后再评估是否引入。
- `Cost of Delay`：适合技术债/平台/合规类问题，第二阶段再按场景补入。
- `OST`：适合更重的持续发现流程，等路由层稳定后再接入。

## 统一模板建议

第一阶段新增的每个模型 reference 统一采用以下章节模板：

1. 目的
2. 什么时候用
3. 不适合什么时候用
4. 核心问题
5. 最小判断步骤
6. 推荐输出
7. 反模式

这样可以确保不同模型之间风格一致，便于 Agent 路由时快速读取。

## 风险与注意事项

### 风险 1：方法库膨胀，主流程失焦

- 表现：`w01` 变成方法论百科，Agent 不知道先做什么。
- 控制：坚持 `1 个主视角 + 1 个补充视角`；第二阶段模型暂不进入主入口。

### 风险 2：模型之间概念重叠

- 表现：`ROI / 价值密度 / RICE / Kano` 混成一套重复排序。
- 控制：明确职责分层，排序统一回到 `ROI + 价值密度`。

### 风险 3：过度模板化限制模型发挥

- 表现：每次都机械套模板，输出僵硬。
- 控制：主文件写路由与原则，模型文档只写透镜和边界，不写过强 prompt。

### 风险 4：文档数量增多导致维护负担上升

- 表现：多个 references 漂移、互相重复。
- 控制：统一模板、单层引用、同一规则只维护一处。
