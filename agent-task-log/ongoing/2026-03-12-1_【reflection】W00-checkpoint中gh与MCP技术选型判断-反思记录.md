## 对话要点沉淀（精简版）

1. `W00` 当前职责非常窄，只负责 GitHub Issue 的“存档/读档”。
2. 在这个边界下，`gh` 已经能直接、稳定、低成本地完成目标。
3. `MCP` 不是不能做，而是当前没有形成足够的复杂度收益比。
4. 技术选型不应围绕“新不新”，应围绕“是否更贴合当前核心价值与约束”。
5. 更稳妥的路径不是立即替换，而是保留语义层稳定，未来按证据决定是否补充新的实现层。

## 关键判断依据

### 1. 先看核心目标，而不是先看可用技术

- `W00` 的核心目标不是“拥有一个更先进的工具协议层”。
- `W00` 的核心目标是“高频、稳定、可恢复地对 GitHub Issue 执行 checkpoint 存档与读档”。
- 若新方案不能显著提高这一目标的达成率，就不应仅因技术新颖而引入。

### 2. 当前 `gh` 方案的优势是直接且可验证的

- GitHub 原生场景贴合：`gh issue list/view/comment` 与 `W00` 的动作几乎一一对应。
- 认证与权限路径成熟：无需额外维护新的服务边界。
- 调试成本低：命令可直接复现，失败时可快速定位是鉴权、参数还是网络问题。
- 认知负担低：对单仓库、单 Agent、本地工作流而言，执行链路短且透明。

### 3. 当前引入 `MCP` 的收益不够大

- 会增加运行时与协议层复杂度。
- 会引入新的服务发现、权限配置、工具封装和调试路径。
- 在只操作 GitHub Issue 且调用模式简单的场景下，这些复杂度目前没有换来决定性收益。

## 为什么这次不该“为了能做而做”

### 教训 1：技术先进性不等于场景适配性

- `MCP` 的价值更多体现在“统一能力暴露、跨客户端复用、结构化工具接口”等层面。
- 但 `W00` 当前并不以这些能力为主要瓶颈。
- 因此，“MCP 可以做”并不自动推出“现在就应该做”。

### 教训 2：复杂度准入必须看证据

- 若没有多客户端复用、跨后端接入、统一权限审计、结构化返回等明确需求，升级基础设施的理由不足。
- “以后可能会用到”不能作为当前增加复杂度的充分理由。

### 教训 3：保留演进空间，比提前抽象更重要

- 现在最合理的做法不是把 `gh` 全量替换成 `MCP`。
- 更合理的是保持 `W00` 的行为契约稳定，把 `gh` 视为默认实现。
- 等未来出现明确信号，再把 `MCP` 作为可选 provider 补进来。

## 何时才值得考虑引入 MCP

以下信号出现时，再评估迁移或补充 `MCP` 实现更合理：

1. 不止一个 Agent 或客户端需要稳定复用同一套 checkpoint 能力。
2. 需要结构化返回结果，避免大量依赖 shell 文本解析。
3. 需要同时接入 GitHub 之外的后端，例如 GitLab、Jira、Notion。
4. 需要统一的权限治理、服务端封装和跨环境能力暴露。
5. 当前 `gh` 方案已经显著阻碍扩展性、可维护性或协作效率。

## 推荐的演进策略

### 当前阶段

- 保持 `gh` 为 `W00` 默认实现。
- 继续用最短路径完成 checkpoint 存档/读档。
- 不为未来假设提前引入 MCP 基础设施。

### 未来阶段

- 若确实出现复用与扩展需求，优先做“实现层并存”而不是“直接替换主路径”。
- 即：语义层继续定义 `save/resume`，执行层允许 `gh` 与 `MCP provider` 并存。
- 这样既保留当前低复杂度优势，也保留后续演进空间。

## 快速判定法：遇到新技术时是否该上

每次技术选型前，先问以下四个问题：

1. 它是否直接强化当前核心目标，而不是仅仅显得更通用？
2. 它带来的收益，是否大于新增的运行时、维护和认知成本？
3. 现有方案是否已经出现明确瓶颈，而不是“理论上可能不够优雅”？
4. 新复杂度能否被局部封装，而不是扩散到整个工作流？

若这 4 问中多数答案是否定的，应优先保留当前简单方案。
