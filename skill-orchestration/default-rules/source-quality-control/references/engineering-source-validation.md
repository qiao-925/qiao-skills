# 工程场景来源验证

> 这份文档用于编码、API、SDK、版本迁移、开源库接入与维护状态判断。目标不是“多看资料”，而是用最接近真实行为的来源，减少猜测式实现和错误接入。

## 1. 何时读取

命中以下任一情况时，应读取本文件：

- 新接第三方库、SDK、框架或服务
- 参数名、方法名、返回值、配置项不确定
- 版本升级、迁移、弃用、兼容性判断
- 报错明显指向 API 使用方式不对
- 评估一个开源库是否值得接入、是否还在维护、真实行为是否可靠

## 2. 工程来源优先级

### API / SDK / 框架行为

优先顺序：

1. 官方文档 / API reference
2. 官方 release notes / migration guide / changelog
3. 官方示例 / 官方仓库 README / 设计说明
4. maintainer comment / 高信号 issue / discussion
5. 社区教程或第三方分析

适用原则：

- 方法签名、参数语义、返回值结构以 `P0` 为准。
- 当文档落后、示例过时或行为不一致时，优先看 release notes 与 maintainer 原始说明。
- 只有在 `P0` 解释不了真实行为时，才用 issue / discussion 补充。

### 开源库接入与维护状态

优先顺序：

1. 官方仓库 README / docs
2. release notes / changelog / roadmap
3. maintainer comment、design note、high-signal issue
4. 社区总结与第三方文章

检查重点：

- 最近 release、commit、issue 响应是否正常
- 是否有明确迁移说明与版本边界
- README 与当前版本行为是否一致
- 是否存在长期未处理的关键缺陷、已知弃用或维护者放弃信号

## 3. 最低执行流程

### Step 1：先收敛未知点

先明确当前到底在查什么，不要一边搜索一边发散：

- 是查 API 用法、参数、返回值，还是查版本差异？
- 是判断“能不能用”，还是判断“为什么出错”？
- 是查文档语义，还是查真实维护状态？

### Step 2：补全最小上下文

至少补齐：

- 库 / 服务名称
- 版本号或版本范围
- 报错信息或目标行为
- 当前使用方式或候选方案

若这些信息缺失，先收集再查来源。

### Step 3：先读 P0，再补高信号经验材料

- 先读官方文档、API reference、release notes。
- 若 `P0` 已足够解释问题，不再扩大搜索面。
- 若 `P0` 与实际行为冲突，再补 maintainer comment 或高信号 issue。

### Step 4：结论必须绑定版本与来源

输出时至少写清：

- 你依据的是哪一类来源
- 结论适用于哪个版本或条件
- 若仍有未验证假设，显式写出

### Step 5：实现后验证真实行为

工程结论不能停留在“来源上看起来对”，应尽量补一个真实验证：

- 最小复现
- 单元测试 / 集成测试
- 运行日志或实际行为检查

如果无法验证，必须降低结论力度。

## 4. 关键判断门槛

### 何时只看文档就够

满足以下条件时，通常只看 `P0` 即可：

- 当前问题只涉及公开 API 的标准用法
- 官方文档和示例一致，没有版本冲突迹象
- 没有社区广泛报告的异常行为

### 何时必须看 release notes 或 migration guide

命中任一条件时，应补看：

- 升级了版本
- 出现弃用或迁移提示
- 示例代码与当前调用方式冲突
- 报错像是版本差异而不是单纯拼写错误

### 何时必须看 maintainer 或高信号 issue

命中任一条件时，应补看：

- 官方文档明显过期
- 文档未解释真实行为或边界条件
- 社区里出现同类问题且 maintainer 已直接回应
- 你正在判断一个库是否值得长期接入，而不是只修一个小问题

## 5. 输出契约

### 编码 / 修复类输出至少包含

- 结论：当前推荐的写法或修复方向
- 来源依据：文档、release notes、maintainer comment 等
- 版本条件：适用于哪个版本或范围
- 风险或例外：什么情况下可能失效
- 验证状态：已验证 / 待验证

### 开源库判断类输出至少包含

- 当前判断：可用 / 谨慎接入 / 不推荐 / 继续观察
- 主依据：维护状态、版本稳定性、真实行为信号
- 主要风险：兼容性、维护性、文档滞后、社区支持等
- 条件化建议：在什么前提下可转为推荐

## 6. 常见反模式

- 靠模型记忆猜 API 参数名或调用方式
- 只看 README，不看 release notes 和 migration guide
- maintainer 已经说明弃用，但仍沿用旧示例
- 用 star 数或热度替代维护状态判断
- issue 里有一个负面案例，就直接推断整个库不可用
- 改完代码不验证，就把来源层面的正确性当成运行层面的正确性
