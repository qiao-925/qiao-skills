# 研究与综合场景来源整合

> 这份文档用于资料收集、技术调研、产品与方案比较、研究总结、长文档写作，以及多 agent 并行证据整合。目标不是“搜更多”，而是把高质量来源转成结构清晰、可追溯、可复查的结论。

## 1. 何时读取

命中以下任一情况时，应读取本文件：

- 做信息收集、技术调研、研究总结
- 产出结构化文档、分析报告、学习材料
- 比较多个产品、工具或方案
- 需要用 multi-agent 并行收集资料，再统一编排终稿

## 2. 研究来源优先级

### 资料收集 / 技术调研

优先顺序：

1. 官方材料、原始数据、原论文、标准文本
2. 权威机构报告、顶级研究论文、核心维护者说明
3. 高质量综述和二级分析

适用原则：

- 关键判断优先由 `P0` 支撑。
- 解释性较强的判断，优先由 `P0 + P1` 交叉验证。
- `P2` 可补充视角，但不能主导结论主线。

### 产品 / 工具 / 方案比较

优先顺序：

1. 官方文档、产品说明、限制页、价格页、更新日志
2. 官方博客、发布会、工程文章
3. 高可信第三方评测或横评

适用原则：

- 不只看功能页，要一起看限制、价格、更新记录。
- 若比较的是“当前能力”，必须优先核对最新来源。
- 第三方横评只能补充，不应代替原始能力说明。

## 3. 文档合成流程

### Step 1：先定主线，不先堆材料

在收集前先明确：

- 这份文档要回答什么核心问题
- 输出类型是什么：结构认知、演化分析、方案比较还是研究综述
- 最终读者最需要带走哪几个判断

如果主线没定，收集再多资料也只会变成拼盘。

### Step 2：按来源等级收集证据

推荐至少做一个简单的证据表：

- 结论或主题
- 主支撑来源等级
- 补充来源等级
- 是否属于推断

这样做的目的不是繁琐，而是防止：

- 关键结论被二手材料偷偷主导
- 多个结论共用一个模糊来源
- 推断与事实混写

### Step 3：先组织判断，再组织章节

正文应该围绕判断组织，而不是围绕来源顺序组织。

优先顺序应是：

1. 先确定主结论和结构主线
2. 再把证据填到对应章节
3. 最后再补引用来源和边界说明

如果按“我看了什么资料”来写，通常会导致主线松散。

### Step 4：区分事实、推断与建议

默认区分三类内容：

- 已验证事实
- 基于资料的推断
- 当前建议或综合判断

当某段是推断时，必须显式写明“这是基于资料的推断”或等价表达。

### Step 5：统一终稿，不直接拼接并行结果

多 agent 并行收集时，worker 应提交：

- 证据包
- 章节草稿
- 修订建议

不应让多个 agent 直接并行写同一份完整终稿。终稿必须由主控统一完成：

- 去重
- 校准术语
- 合并冲突证据
- 调整结论力度
- 保证主线连贯

## 4. 文档输出契约

当任务产出资料型文档、分析报告、研究总结时，默认采用：

1. 目录
2. 文档正文
3. 引用来源

### 正文最低要求

- 逻辑连贯、可读性强、有明确主线
- 关键结论能回指来源
- 不把材料卡片直接堆成章节
- 强结论、较新结论优先交叉验证
- 语言精炼，突出重点；若有硬性限制，默认不超过 300 行

### 比较类输出最低要求

- 比较维度明确，而不是只给总体印象
- 不同结论依赖不同前提时，写成条件化结论
- 若使用第三方横评，指出其局限和来源等级

## 5. 并行执行与可重复执行

### 并行执行前提

- 任务能按主题、来源域、时间线、子问题拆分
- 已先定义来源等级和输出结构
- 已明确每个 worker 的 ownership

### 并行执行默认做法

1. 主控先定义边界、目录、来源等级和证据标准
2. worker 并行收集不同主题或来源域的材料
3. 统一回收证据包，而不是直接拼接正文
4. 主控统一编排、去重、补过渡、校准结论力度

### 可重复执行最小清单

每次执行至少保留：

- 任务目标
- 检索范围或主题边界
- 主要来源等级
- 关键来源链接
- 纳入 / 剔除理由
- 执行时间点
- 明确事实与推断的边界

## 6. 常见反模式

- 把来源越多等同于质量越高
- 用二手总结主导整篇调研结论
- 强结论没有 `P0` 支撑，也不做交叉验证
- 把第三方横评当作原始事实
- 多个 agent 并行收集后直接拼接终稿
- 文档有很多信息，但没有主线、没有独特判断、没有来源回指
