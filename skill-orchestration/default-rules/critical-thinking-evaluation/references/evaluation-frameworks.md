# 批判性评估方法骨架

> 这份文档不是让用户照表作答，而是提供给 Agent 在内部执行批判性评估时使用的方法骨架。只有在关键信息缺失时，才把最必要的问题外显给用户。
>
> 在信息极易获取的时代，稀缺的正是对世界保持结构化思考的习惯、对一切确定保持怀疑的精神以及对真理的好奇心。

形成背景：

- Redis vs ElasticSearch 技术分析优化过程
- Cursor 并行冲突的附和陷阱案例
- 苏格拉底式提问策略

这份方法骨架的目标，是为后续技术分析提供一个可复用的综合批判性思维框架。

## 1. 要素检查

适用：

- 技术选型
- 架构判断
- 方案比较
- 优先级排序

至少检查六项：

1. 目标：当前到底要解决什么问题。
2. 证据：已知事实、数据、约束、经验依据是什么。
3. 假设：哪些前提尚未被验证。
4. 替代方案：还有什么可行路径。
5. 后果：当前建议失败时会造成什么代价。
6. 结论力度：当前是确定结论、倾向建议，还是暂定判断。

如果六项里有两项以上明显空缺，且会改变建议方向，应降低结论力度，必要时只外显一个关键问题，而不是靠语气硬撑结论。

## 2. 关键追问

适用：

- 证据不足
- 方案分叉明显
- 用户要求高代价判断

优先追问的顺序：

1. 目标有没有说清。
2. 约束有没有说清。
3. 当前判断依赖的关键假设是什么。
4. 是否存在更简单或更低风险的替代方案。
5. 这个结论最可能在哪里失效。

对外提问时，只选当前最关键的一问，不要把内部检查表整套抛给用户。

## 3. 结构化论证

适用：

- 需要给出明确建议
- 需要说服用户接受某个方向
- 需要解释为什么不推荐某个方案

最小结构建议：

1. 主问题：当前到底在判断什么。
2. 结论或主张：当前倾向是什么。
3. 依据：支撑结论的关键事实和约束。
4. 假设：哪些前提尚未验证。
5. 推理：为什么这些依据能推出这个结论。
6. 限定：这个结论在什么条件下成立。
7. 反方或反例：在什么情况下应改选别的方案。

如果问题较轻，可以压缩成“结论 + 依据 + 推理”。如果问题较重，应至少补上“假设”“限定”和“反方”。

如果一个分析只是不断换结论，却没有解释上述结构中哪一环变了，这通常不是严谨修正，而是附和式摆动。

## 4. 魔鬼代言人

适用：

- 当前分析看起来过于顺滑
- 证据来源单一
- 用户和 Agent 明显已经对某个方向形成情感倾向
- 方案代价较高，但反方分析仍然很弱

执行方式：

1. 先假设当前分析可能是错的。
2. 从三类错因倒推：
   - 数据或证据错了
   - 推理链错了
   - 边界条件漏了
3. 检查这些错因里，哪些已经被当前分析处理，哪些还没有。

可直接使用的最小模板：

```text
为什么这个分析可能是错的：
- 证据可能错在 [...]
- 推理可能错在 [...]
- 边界可能错在 [...]
```

## 5. 失败倒推

适用：

- 高成本投入
- 大范围重构
- 架构迁移
- 技术路线判断

执行方式：

1. 先假设当前建议已经执行。
2. 再假设它在后续失败了。
3. 倒推最可能的三个失败原因。
4. 检查这些失败原因是否已经被当前建议处理。

如果失败点没有被处理，就不应把当前建议表述成高确定性的最优解。

## 6. 附和陷阱警示案例

### Cursor 并行冲突的附和陷阱

- 问题背景：讨论 Cursor 的对话模式和 Background Agent 是否冲突。
- AI 第一次回答（错误）：说“有冲突，不能同时运行”，这是基于推测。
- AI 第二次回答（正确）：承认“实际使用中确实没有冲突”，这是基于用户体验。
- 警示意义：AI 会基于有限信息推测；当用户质疑时，如果不回到证据和假设检查，就会立即“附和”用户观点。

这个案例应当提醒 Agent：

1. 被用户质疑，不等于当前判断已经被证伪。
2. 判断要修正，必须说明是什么证据、经验或前提变化导致修正。
3. 如果前后结论不同，却说不清结构中哪一环变了，就要警惕自己是在附和，而不是在分析。

## 7. 默认输出模板

可直接用于重要判断：

```text
主问题：
- 当前要判断的是 [...]

结论：
- 我当前倾向于 [方案]。

依据：
- 已知目标是 [...]
- 当前关键约束是 [...]
- 主要证据是 [...]

假设：
- 这个判断默认 [前提] 成立。

反方：
- 如果 [反例或边界] 成立，则当前结论需要调整。

替代方案：
- 另一个可行方案是 [...]，但它更适合在 [条件] 下采用。

风险：
- 当前方案最可能失败在 [...]

下一步：
- 若要提高判断可靠性，下一步最值得补的是 [...]
```

## 8. 与立场偏见分析联动

当分析对象本身带有明显传播立场、机构利益、公共争议或价值倾向时，仅靠通用批判性骨架还不够，还应继续补做：

- 立场倾向识别
- 利益相关方分析
- 缺失视角检查
- 偏见修正与中性重述

这部分请继续读取 `bias-and-stance-analysis.md`，不要和来源等级、时效性、事实核验混写。
