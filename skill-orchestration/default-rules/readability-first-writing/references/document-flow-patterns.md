# 文档阅读流优化模式

## 1. 标题要像路标，不要像装饰

标题的职责是告诉读者：

- 这一段在讲什么
- 它和上一段是什么关系
- 读完这一段会得到什么

好标题优先清楚，其次才是风格。

## 2. 段首引导句

长段或复杂段落，建议在段首先给一句引导：

- 这一段的主问题是什么
- 为什么现在讲这一段
- 它和前后文如何衔接

段首引导句能显著降低读者的进入成本。

## 3. 长文档开头先给高密度概括

当内容较长、判断较多、层次较深时，开头不应直接进入铺垫或过程展开，而应先给一个高信号概括。

这个概括的目标是让读者在最短时间内知道：

- 这份内容在解决什么问题
- 核心判断或主张是什么
- 后文会按什么结构展开
- 读者接下来该带着什么预期往下读

可用的高密度概括模式包括：

- `定义 + 命题 + 架构 + 路径`
- `问题 + 判断 + 结构 + 下一步`
- `结论 + 原因 + 展开顺序`

其中 `定义 + 命题 + 架构 + 路径` 适合规划文档、架构文档、研究总结等高层判断型长文档：

- 定义：先说当前对象是什么，边界在哪里
- 命题：再说这份文档的核心主张或冻结判断
- 架构：说明内容将按什么骨架组织
- 路径：说明接下来按什么顺序推进

它是强模式，不是硬模板。如果文档类型不适合，不要机械套用字面结构，但应保留“先给读者总览，再展开细节”的原则。

## 4. 列表只在扫描收益明显时使用

适合列表的内容：

- 同级要点
- 检查项
- 步骤序列
- 对比维度

不适合列表的内容：

- 本来就需要连续论证的长推理
- 强依赖上下文承接的细腻解释
- 只是为了让页面“看起来更丰富”

## 5. 表格与引用块的使用边界

### 表格

适合：

- 规则对照
- 方案比较
- 场景与要求映射

不适合：

- 塞长段正文
- 用表格替代本应展开解释的逻辑

### 引用块

适合：

- 原文摘录
- 重点提醒
- 需要视觉停顿的小结

不适合：

- 整篇到处都是引用块
- 只是为了“显眼”而滥用

## 6. 视觉元素应克制

emoji、分隔线、符号装饰都不是禁用，但默认要克制。

判断是否该用的标准：

- 是否真的帮助扫描
- 是否与当前语境匹配
- 是否会分散注意力
- 是否会削弱专业感

如果只是为了“更好看”，通常不值得加入。

## 7. 风格友好化的正确方向

更稳定的友好写法通常来自：

- 少一点僵硬公文腔
- 多一点直接解释
- 少一点套话
- 多一点“为什么这件事重要”

而不是来自：

- 强行玩梗
- 强行网络化
- 强行口语化
- 强行做成“活泼文案”

## 8. 交付前快速复核

交付前快速看五件事：

1. 目录或标题能不能带路
2. 长文档开头有没有先给高密度概括
3. 段首有没有给读者入口
4. 重点是不是埋在中后部
5. 格式元素是不是帮忙，而不是添乱
