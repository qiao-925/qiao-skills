# 知识综合

> 这个 skill-unit 处理的是“如何把资料变成高价值内容”问题。它不负责决定资料是否可信，那是 `source-quality-control` 的职责；它负责把可信材料组织成有主线、有因果、有洞察的输出。

## 核心原则

### 1. 历史优先 —— 有演化线索时，先讲前因后果

- 当对象存在明显演化脉络时，优先通过时间线或阶段变化解释它为什么变成今天这样。
- 历史不是背景点缀，而是帮助读者理解设计结果的因果链。
- 如果对象几乎没有历史演化价值，不必硬讲时间线。

### 2. 设计意图显式化 —— 解释目标、约束与取舍

- 高价值内容不止说“它是什么”，还要解释“为什么这样设计”。
- 设计目标、约束条件、关键取舍、阶段性妥协，往往比表面结构更能解释本质。
- “思想”与“哲学”如果不能落到真实决策背景，就不应空泛悬浮。

### 3. 阅读路线前置 —— 先告诉读者怎么读

- 开篇应给出思维路线梗概：主线是什么、因果关系是什么、重点结论在哪里。
- 读者应在进入正文前知道这篇内容将如何展开。
- 如果开头没有阅读路线，后文再丰富也容易变成材料迷宫。

### 4. 速查信息下沉 —— 让重点可快速回看

- 对复杂主题，允许提供速查表、摘要框、要点清单等附属产物。
- 速查信息的职责是帮助读者快速形成 sense，而不是替代正文主线。
- 速查层应服务正文，而不是抢走正文的位置。

### 5. 专注价值 —— 洞察优先于材料数量

- 研究型内容的目标不是“收集更多”，而是“让读者更快抓住本质”。
- 观点和结论必须有证据支撑，但不能止步于堆证据。
- 优先产出有解释力、有取舍感、有层次的洞察，而不是平铺罗列。

## AI Agent 行为要求

### 默认适用场景

| 场景 | 最低要求 | 不该做什么 |
|------|----------|------------|
| 技术调研文档 | 把资料整理成主线、因果和结论 | 只按来源顺序堆材料 |
| 宏观理解文档 | 解释结构、演化与关键判断 | 只有知识点罗列，没有解释框架 |
| 专题分析 | 给出问题背景、推导路径和洞察 | 上来就给结论，不交代来路 |
| 基于旧稿优化 | 保留正确内容，重构主线与表达 | 为了改动量机械重写全部内容 |

### 默认执行方式

1. 先判断当前任务是“从零综合”还是“基于已有内容继续优化”。
2. 在开始写正文前，先确定主线、因果关系和阅读路线。
3. 对存在演化脉络的对象，优先组织时间线和关键转折。
4. 对关键结构、机制或阶段性变化，补上设计目标、约束和取舍解释。
5. 如果对象复杂，补一层速查信息，但不让它替代正文。
6. 最终输出以洞察、层次和主线完整性为先，而不是以材料数量为先。

### 多 Agent 组织方式

- 可以并行收集材料，但不要放养式并行写终稿。
- 更推荐：主控收敛 + 分工采集 + 最终统一编排。
- 收敛要克制，但不能没有；应采用“最小必要收敛”，而不是完全放任。

### 与其他 skill 的协同边界

- 与 `source-quality-control`：先保证来源质量，再进行知识综合。
- 与 `general-understanding-research`：当交付目标明确是“结构视角 + 演化视角”两份主产物时联动。
- 与 `critical-thinking-evaluation`：当需要明确区分事实、推断和开放问题时联动。

## 判断标准

- 是否在开篇给出了清晰的阅读路线和重点结论。
- 是否解释了前因后果，而不是只罗列当前状态。
- 是否把设计目标、约束和取舍讲清楚，而不是空谈哲学。
- 是否有速查层帮助快速形成认知，但没有破坏正文主线。
- 是否产出了洞察，而不是来源堆叠或知识卡片拼盘。

## 反模式

- 把研究文档写成资料摘抄集。
- 只说“是什么”，不说“为什么重要”。
- 强行讲历史，结果只有年份没有因果。
- 把“思想”“哲学”写成空泛口号，没有决策上下文。
- 用速查表替代正文，导致主线消失。
- 为了并行而牺牲统一术语、叙事顺序和结论力度。
