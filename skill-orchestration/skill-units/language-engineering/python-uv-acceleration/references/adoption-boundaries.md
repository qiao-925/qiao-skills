# 采用边界：`uv` 适合替代什么，不适合替代什么

## 核心判断

`uv` 最适合替代的是：

- `pip`
- `venv`
- `virtualenv`
- 以 `requirements.txt` 为核心的轻量依赖工作流

`uv` 不是一个“所有 Python 项目都必须切换”的口号。

## 优先采用 uv 的场景

- 新建脚本型、工具型、小中型 Python 项目
- 现有仓库本来就是 `pip + venv`
- 用户明确希望提升依赖安装速度
- CI 或本地初始化主要瓶颈在环境创建与依赖安装

## 不应强推 uv 的场景

- 项目已稳定使用 `poetry`、`pdm`、`hatch`
- 团队已有成熟工具链、锁文件与发布流程
- 当前任务只是修 Python 代码，不涉及环境与依赖
- 迁移成本明显高于当前收益

## 推荐表达方式

### 适合直接替换

```text
当前项目是典型 pip/venv 工作流，可以直接改用 uv 来加速。
```

### 适合保守处理

```text
当前项目已使用 poetry 管理依赖，不建议直接把 uv 设为主工具链。
如果只是追求安装速度，可以先评估局部替换是否值得。
```

## 检查清单

- [ ] 是否确认当前项目不是另一套稳定管理器主导。
- [ ] 是否明确这次任务真的和环境/依赖工作流相关。
- [ ] 是否把“默认推荐”与“强制迁移”清楚地区分开。
