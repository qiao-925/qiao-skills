# 类型边界：优先标清公共接口

## 核心目标

类型提示的价值在于减少边界歧义，而不是追求“注解覆盖率”。

## 优先级

### 第一优先级

- 公共函数与公共方法
- 跨模块调用边界
- 返回值不显然的函数
- 容易混淆的数据结构

### 第二优先级

- 关键类属性
- 复杂回调与高阶函数
- 容易误用的可空值与联合类型

### 低优先级

- 含义非常直接的短生命周期局部变量
- 仅为“看起来完整”而添加的重复注解

## 默认基线

- 有返回值就写清返回类型。
- 无返回值显式写 `-> None`。
- 新增或修改公共边界时，优先补齐类型提示。
- 复杂公共 API 在行为不显然时，补必要 docstring。

## 风格建议

- 尊重项目当前 Python 版本和 typing 风格。
- 如果项目已普遍使用 `list[str]` 这类现代写法，沿用它。
- 如果项目仍使用 `List[str]`、`Optional[str]` 等传统写法，先保持一致，不要局部切风格。

## 反模式

- 为了“规范”给每个局部变量都加注解。
- 不看项目 Python 版本，直接引入不兼容的类型语法。
- 只补参数类型，不补返回值，让边界仍然模糊。

## 检查清单

- [ ] 公共边界与跨模块接口类型是否清楚。
- [ ] 类型写法是否与项目现有风格一致。
- [ ] docstring 是否只补在真正需要解释的地方。
