# 命名与结构：顺着 Python 习惯读代码

## 命名基线

| 对象 | 推荐形式 |
|------|----------|
| 模块文件 | `snake_case.py` |
| 包目录 | `snake_case/` |
| 类 | `PascalCase` |
| 函数 / 变量 | `snake_case` |
| 常量 | `UPPER_SNAKE_CASE` |
| 私有成员 | `_snake_case` |

## 命名原则

- 表意优先，缩写克制。
- 同一概念尽量用同一术语，不混用近义词。
- 布尔值优先使用 `is_`、`has_`、`can_`、`should_` 这类前缀。

## 导入与文件结构

- 导入通常分为：标准库、第三方库、本地模块。
- 组间空行，组内保持稳定顺序。
- 避免通配符导入。
- 跨包引用优先使用项目当前主流方式；若无明确约定，优先清晰的绝对导入。

## 函数与模块结构

- 函数优先保持单一主路径。
- 错误和边界情况前置，减少嵌套。
- 一个模块只承载一组紧密相关的职责。
- 如果读者需要跳很多层才能看懂主流程，通常说明结构过重。

## 反模式

- 用 `calc`、`proc`、`tmp` 这类模糊缩写承载长期语义。
- 一个文件同时塞进不相关职责。
- 导入顺序和分组混乱，增加阅读成本。
- 为了“优雅”拆成很多跳转，主路径反而难追。

## 检查清单

- [ ] 名称是否能直接表达主要意图。
- [ ] 术语在同一模块内是否保持一致。
- [ ] 导入分组是否稳定、清楚。
- [ ] 主流程是否可以顺序阅读。
