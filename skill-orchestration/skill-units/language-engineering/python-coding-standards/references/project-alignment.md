# 项目对齐：先读现状，再落规范

## 核心问题

Python 规范最常见的误用，不是“标准不够”，而是“无视仓库现状”。

真正有价值的实现基线，应该先判断：

- 项目已经有哪些明确约定
- 哪些约定是稳定的
- 哪些地方根本还没有约定，需要这次补默认值

## 建议检查顺序

1. 读取 `pyproject.toml`、`ruff.toml`、`mypy.ini`、`pytest.ini` 等配置文件。
2. 查看相邻模块与近期代码，确认命名、导入、logger、类型、测试的现状。
3. 找出“已有约定”与“尚未收敛”的边界。

## 应优先沿用的内容

- 现有格式化与 lint 工具链
- 现有日志设施或框架约定
- 现有类型风格
- 现有目录与导入习惯

## 可以用通用基线兜底的内容

- 新增模块的命名
- 没有统一约定时的导入分组
- 新增公共接口的类型提示
- 没有日志封装时的默认日志方式

## 不该做的事

- 仅因自己更喜欢另一套风格，就重写整个文件
- 忽略仓库约定，直接套用通用模板
- 把某个项目中的私有实践上升为“所有 Python 项目都应如此”

## 决策规则

- 仓库已有稳定约定：沿用。
- 仓库存在多套并行写法：优先贴近当前文件周边。
- 仓库没有明确约定：使用本 skill 默认基线。

## 检查清单

- [ ] 已查看项目配置或相邻代码，而不是凭印象改写。
- [ ] 这次变更优先贴合当前文件所在区域的风格。
- [ ] 没有把项目私货误写成通用 Python 规范。
