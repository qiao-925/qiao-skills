# 实现级：代码直达，主路径清晰

## 核心问题

实现层最常见的问题不是“不会写”，而是“写得太绕”。

实现级简化关注两件事：

- 代码是否直达业务意图
- 主路径是否一眼能读出来

## 代码级简化规则

- 能用命名说清楚，就不要再加解释性层级。
- 能复用已有结构，就不要并行造新模式。
- 能顺序写清楚，就不要提前抽象成复杂框架。
- 代码的第一职责是让人理解当前行为，而不是展示抽象技巧。

## 函数级简化规则

- 一次只完成一个明确目标。
- 优先线性主路径，减少条件嵌套。
- 例外与错误路径前置，主流程保持连续。
- 参数过多、职责混杂、流程编排与业务计算混写时，应拆分。

## 常见收敛动作

- 用 guard clauses 减少嵌套。
- 把重复分支提取为小函数。
- 去掉收益不明的适配层、配置层、包装层。
- 让核心流程能在短路径内读完。

## 警示信号

- 一个函数同时做校验、计算、IO 和状态编排
- 一个文件靠大量注释才能解释清楚流程
- 为了“优雅”拆成很多跳转，导致主流程难追
- 新增抽象后，阅读路径反而更长

## 检查清单

- [ ] 核心流程是否可以顺序阅读。
- [ ] 新增抽象是否有明确复用或隔离价值。
- [ ] 函数名和文件名是否准确表达主意图。
- [ ] 改动后是否比改动前更容易理解，而不是更“高级”。
