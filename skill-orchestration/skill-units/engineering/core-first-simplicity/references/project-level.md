# 项目级：主亮点先行，范围随后

## 核心问题

项目层最容易失控的地方，不是能力不够，而是目标太散。

项目级复杂度控制要先回答：

- 用户最终会因为什么记住这个项目
- 当前迭代最值得验证的闭环是什么
- 哪些需求只是“看起来也不错”，但不该进入本轮

## 必问三题

1. 如果当前版本只能保留一个亮点，这个亮点是什么？
2. 本次新增内容是否直接强化这个亮点？
3. 不做这件事，核心价值是否真的受损？

## 决策规则

- 直接强化主亮点：优先做。
- 能形成闭环支撑，但不是亮点本身：做最小版本。
- 只有展示价值，没有闭环价值：延后。
- 会显著分散认知与实现精力：删除。

## 常见简化动作

- 合并相近需求，减少并行路线。
- 砍掉“为了显得完整”而补的边缘功能。
- 把高成本优化挪到核心闭环验证之后。
- 把“未来路线图”从当前实现里抽出去，单独记录。

## 输出格式建议

当需要对用户解释时，可按下面格式收敛：

```text
核心亮点：
- ...

本轮必须做：
- ...

本轮延后：
- ...

本轮删除：
- ...
```

## 检查清单

- [ ] 当前迭代目标可以一句话描述。
- [ ] 本轮工作项大多能映射到同一主亮点。
- [ ] 延后项是显式的，而不是隐含在讨论里。
- [ ] 没有为了“看起来完整”而挤压主线交付。
