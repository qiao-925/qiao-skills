# 架构级：复杂度预算与结构收敛

## 核心问题

架构层的复杂度一旦引入，回收成本往往很高。

因此，架构级简化不是“反对设计”，而是要求每一层复杂度都必须说明：

- 为什么现在就需要
- 不加会坏在哪里
- 复杂度会被封装在什么边界内

## 复杂度预算准入

只有满足下列至少两项，才值得认真考虑新增架构复杂度：

- 有明确性能、稳定性、容量或安全证据
- 简单方案已验证不够用
- 新复杂度可以被局部封装，不扩散为全局负担
- 用户目标明确需要该扩展能力，而不是猜测未来可能需要

## 架构收敛动作

### 1. 主路径单线化

- 优先保留一条最清晰、最容易验证的主链路。
- 非主路径能力尽量做可选分支，而不是全局必经层。

### 2. 扩展点延后

- 插件化、策略化、平台化都应在真实需求出现后再做。
- 如果只有一个实现，优先清晰实现，不急着抽象。

### 3. 依赖数量收缩

- 减少横向依赖和中间层。
- 避免为了“解耦”增加更多需要维护的抽象。

### 4. 模块边界按稳定变化面划分

- 总是一起变化的内容可以内聚。
- 变化频率不同、演化方向不同的内容才值得拆开。

## 高风险反模式

- 需求未稳先建通用框架
- 为未来规模预埋过多层次
- 只有一个实现却设计出多实现体系
- 把可选能力升级成全局必经路径

## 检查清单

- [ ] 主链路是否能独立解释，不依赖大段背景说明。
- [ ] 新层、新抽象、新接口是否有明确证据支撑。
- [ ] 是否存在当前可删除而不影响主目标的层。
- [ ] 模块边界是围绕稳定职责，而不是围绕想象中的复用。
