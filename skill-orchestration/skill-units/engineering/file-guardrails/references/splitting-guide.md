# 大文件拆分指南

## 触发条件

满足任一条件，就应进入拆分讨论：

- 单文件超过 300 行
- 主路径难以在一次阅读中看清
- 一个文件同时承载多种变化原因
- 需要大量注释才能帮助阅读

## 拆分原则

### 1. 按职责拆，不按行数平均切

正确问题是“哪些内容应该分开演化”，而不是“每个文件切多少行”。

### 2. 先保外部接口稳定

如果文件已被外部使用，拆分时优先维持公共入口不变，再在内部重组。

### 3. 主路径和支撑逻辑分开

- 主流程编排
- 数据校验
- 格式转换
- IO 与外部适配

这些内容适合分开管理。

### 4. 拆分后每个文件仍需可独立解释

如果拆完后每个文件都还在做三四件事，说明拆分只是机械移动代码。

## 拆分流程

1. 标出当前文件的职责块与公共入口。
2. 写出拆分方案：新文件列表、各自职责、预估行数。
3. 决定哪些入口保持兼容，哪些需要迁移。
4. 执行拆分并更新导入。
5. 复核：每个文件更短，也更清楚。

## 常见拆分模式

### 模式 1：主流程 + 支撑能力

- `service.py` 保留主流程
- `validator.py` 负责校验
- `formatter.py` 负责输出转换

### 模式 2：领域职责拆分

- `repository.py` 负责存取
- `service.py` 负责业务决策
- `adapter.py` 负责外部系统对接

### 模式 3：包入口稳定

- `__init__.py` 只暴露公共 API
- 具体实现下沉到多个子文件

## 交付拆分方案时建议包含

```text
拆分原因：
- ...

拆分后文件：
- a.py：...
- b.py：...

兼容策略：
- ...
```

## 检查清单

- [ ] 每个文件都在 300 行以内。
- [ ] 每个文件职责能一句话说清。
- [ ] 外部导入与调用是否保持稳定或有明确迁移方案。
- [ ] 主路径是否比拆分前更容易理解。
