# 顶部说明指南

## 核心目标

顶部说明的目标只有一个：让首次读者在几秒内知道这个文件是做什么的。

它不是用来：

- 讲完整架构
- 讲项目历史
- 逐项复述代码里已经清楚表达的内容

## 推荐结构

### 最小版

- 一句话说明文件用途

### 标准版

- 一句话说明文件用途
- 2-4 个主要职责
- 必要时给一个最短示例

### 复杂文件增强版（按需使用）

- 一句话说明文件用途
- 2-4 个主要职责
- 2-4 个主要组成、阅读入口或核心部分
- 只有在边界容易被误判时，补一句“本文件不负责什么”
- 必要时给一个最短示例

这一版只适用于阅读成本较高、职责边界容易误判、首次进入成本明显偏高的文件。
不要把它当成所有代码文件都要套的统一模板。

## 写法原则

### 1. 功能优先

优先说明“这个文件负责什么”，而不是“这个文件属于哪一层”。

### 2. 首句最重要

第一句话必须能独立成立，读完这一句就应大致知道文件用途。

### 3. 不重复显而易见的信息

如果类名、函数名、文件名已经说清楚，不必在注释里机械重复。

### 4. 不把结构问题转嫁给注释

如果需要很长的头注释才能解释清楚，通常说明文件本身已过重。

### 5. “不负责什么”要谨慎使用

只有在文件边界经常被误读、且一句话能明显减少误判时，才补“不负责什么”。

如果只是为了显得更完整而机械补一句，通常是在制造噪音。

## 应避免的内容

- 大段架构分层说明
- 逐行执行流程讲解
- 设计模式罗列
- 过多方法清单
- 为了凑模板而硬写“本文件不负责什么”
- 已经过时却没人维护的示例

## 更新时机

| 情况 | 动作 |
|------|------|
| 新增核心职责 | 更新头注释 |
| 修改公开入口 | 视情况更新示例 |
| 只做内部重构 | 复核是否仍准确 |
| 只修局部 bug | 通常不需要改 |

## 检查清单

- [ ] 第一行是否清楚说明文件用途。
- [ ] 如果是复杂文件，是否补充了最有助于首次阅读的主要组成或阅读入口。
- [ ] “不负责什么”是否真的有助于澄清边界，而不是模板化补齐。
- [ ] 注释是否短于理解成本本身。
- [ ] 是否存在过时信息。
- [ ] 是否夹带过多架构背景。
