# 聚焦原则详细说明

## 1. "最小光辉点"定义

### 1.1 什么是"最小光辉点"

**项目如果只能有一个让人记住的亮点，那是什么？**

示例：
- RAG 系统："智能路由，自动选择最佳检索策略"
- 文档工具："所见即所得的编辑体验"
- CLI 工具："零配置开箱即用"

### 1.2 如何识别

问自己：
1. 别人用一句话怎么介绍这个项目？
2. 如果只能保留一个功能，保留哪个？
3. 去掉什么会让项目失去存在意义？

## 2. 剃刀法则应用

### 2.1 功能评估

每个功能问自己：

| 问题 | 是 | 否 |
|------|----|----|
| 直接强化"最小光辉点"？| ✅ 做 | ❌ 考虑砍 |
| 用户强烈需求？| ✅ 做 | ❌ 延后 |
| 有更简单的方式？| 用简单方式 | - |

### 2.2 砍掉的勇气

可以砍掉的：
- "锦上添花"的功能
- "未来可能用到"的设计
- "别人都有"的功能
- 耗时但非核心的优化

### 2.3 延后的智慧

可以延后的：
- 非核心功能的完善
- UI 美化
- 性能优化（除非是瓶颈）
- 完整的测试覆盖

## 3. "不做什么"清单

### 3.1 本项目不做

- ❌ 不追求功能全面
- ❌ 不过度设计架构
- ❌ 不等待完美方案
- ❌ 不分散精力到非核心功能

### 3.2 如何使用

1. 新功能前，检查是否在"不做"列表
2. 设计决策时，确认不违反"不做"原则
3. 定期回顾，更新"不做"列表

## 4. 实践指南

### 4.1 功能开发

```
1. 这是核心功能吗？
   ├─ 是 → 全力做好
   └─ 否 → 最简方案或延后

2. 有更简单的方式吗？
   ├─ 有 → 用简单方式
   └─ 无 → 评估是否必要

3. 会分散精力吗？
   ├─ 会 → 延后或砍掉
   └─ 不会 → 继续
```

### 4.2 架构设计

```
1. 刚好能用？
   ├─ 是 → 够了
   └─ 否 → 继续简化

2. 为未来设计？
   ├─ 是 → 警惕过度设计
   └─ 否 → 继续

3. 需要重构时再重构
```

### 4.3 质量标准

| 领域 | 核心功能 | 非核心功能 |
|------|----------|------------|
| 功能完整 | ✅ 必须 | ⚠️ 可简陋 |
| 测试覆盖 | ✅ 关键路径 | ❌ 可无 |
| UI 美化 | ⚠️ 可用即可 | ❌ 最简 |
| 文档 | ⚠️ 关键说明 | ❌ 可无 |

## 5. 检查清单

功能评估：
- [ ] 直接服务于核心目标？
- [ ] 有更简单的实现？
- [ ] 是否过度设计？
- [ ] 精力是否被分散？

设计评估：
- [ ] "刚好能用"？
- [ ] 为未来过度设计？
- [ ] 需要时再重构？
